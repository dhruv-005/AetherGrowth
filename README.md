> ### 🧠 What is AetherGrowth OS?
>
> **AetherGrowth OS** is a production-ready, enterprise-grade **AI Orchestration Platform**
> engineered to automate the full lifecycle of digital marketing campaigns — from intelligent
> audience research and multi-channel campaign generation to real-time A/B experimentation,
> performance analytics, and autonomous optimization loops.
>
> The platform is architected on top of Anthropic's **Model Context Protocol (MCP)** —
> a next-generation agentic communication standard — enabling isolated, secure, and
> highly concurrent AI sub-processes to collaborate intelligently across distributed
> memory boundaries with deterministic, low-latency Stdio communication channels.

## 🌐 Overview & Vision

**AetherGrowth OS** redefines how enterprises approach digital marketing automation
by replacing fragmented, manually-operated tool chains with a unified, AI-native
orchestration fabric. The platform enables marketing teams and growth engineers to
deploy fully autonomous campaign pipelines that continuously learn, adapt, and
self-optimize — without requiring constant human intervention.

### 🎯 Core Mission

The fundamental mission of AetherGrowth OS is to collapse the gap between
**AI intelligence** and **marketing execution** by providing:

- **Autonomous Campaign Generation** — AI-driven creation of multi-channel
  marketing campaigns informed by deep market intelligence and audience profiling.

- **Real-Time Swarm Coordination** — Four specialized MCP agents operating
  concurrently across dedicated cognitive scopes, coordinated by a central
  FastAPI supervisor through high-throughput Stdio communication channels.

- **Continuous Experimentation** — Automated A/B and multivariate testing
  pipelines that evaluate statistical significance and surface winning variants
  without manual configuration.

- **Unified Intelligence Dashboard** — A Next.js operational control canvas
  providing real-time telemetry, agent state inspection, and campaign
  performance visualization across all active orchestration pipelines.

### 🔄 Data Flow Architecture

```

        │
        ▼
Next.js Frontend (React Component)
        │
        │ HTTP POST /api/campaigns/create
        ▼
FastAPI Backend (Route Handler)
        │
        ├──── Stdio Write ──→ database-mcp (Persist campaign state)
        │                          │
        │                          └──→ Confirmation ACK
        │
        ├──── Stdio Write ──→ analytics-mcp (Initialize tracking)
        │                          │
        │                          └──→ Tracking ID returned
        │
        ├──── Stdio Write ──→ ab-testing-mcp (Setup experiments)
        │                          │
        │                          └──→ Experiment IDs returned
        │
        └──── Stdio Write ──→ comms-mcp (Broadcast notifications)
                                   │
                                   └──→ Notification ACK

        │
        ▼
FastAPI aggregates all MCP responses
        │
        ▼
JSON Response → Next.js Frontend → Dashboard Update (Zustand Store)
```

## 🤖 MCP Microservices Deep Dive

Each MCP microservice is a fully autonomous **Node.js TypeScript process** that
communicates with the FastAPI backend supervisor exclusively through **Stdio channels**.
This architecture guarantees zero-network-overhead, deterministic latency, and
strict process isolation.

### 🗄️ database-mcp — Long-Term State Management Agent

**Primary Responsibility:** Acts as the authoritative source of truth for all
persistent campaign data, agent memory states, experiment configurations, and
historical performance records.

**Exposed MCP Tools:**

```typescript
// Tool: store_campaign
// Persists a new campaign record to the local state store
{
    name: "store_campaign",
    description: "Persist a campaign record to the database",
    inputSchema: {
        campaign_id: string,
        name: string,
        channel: string,
        audience: object,
        content: object,
        status: "draft" | "active" | "paused" | "completed"
    }
}

// Tool: retrieve_campaign
// Retrieves a specific campaign by its unique identifier
{
    name: "retrieve_campaign",
    description: "Retrieve a campaign record by ID",
    inputSchema: {
        campaign_id: string
    }
}

// Tool: list_campaigns
// Returns a paginated list of all stored campaigns
{
    name: "list_campaigns",
    description: "List all stored campaigns with optional filters",
    inputSchema: {
        status?: string,
        limit?: number,
        offset?: number
    }
}

// Tool: update_campaign_status
// Updates the operational status of an existing campaign
{
    name: "update_campaign_status",
    description: "Update campaign status field",
    inputSchema: {
        campaign_id: string,
        status: "draft" | "active" | "paused" | "completed"
    }
}
```

## 🖥️ Frontend Dashboard Deep Dive

The frontend is built with **Next.js 14+** using the App Router architecture and
**React functional components** with hooks. Global state is managed via **Zustand**,
providing reactive, lightweight state management without the overhead of Redux or
Context Provider trees.

```typescript
    // ── Analytics State ─────────────────────────────────────────

    // ── Actions ──────────────────────────────────────────────────
    setCampaigns: (campaigns: Campaign[]) => void;
    setActiveCampaign: (campaign: Campaign | null) => void;
    setLoading: (isLoading: boolean) => void;
    addNotification: (notification: Notification) => void;
    dismissNotification: (id: string) => void;
    appendLog: (log: LogEntry) => void;
    clearLogs: () => void;
}

export const useCampaignStore = create<CampaignStore>((set) => ({
    campaigns: [],
    activeCampaign: null,
    isLoading: false,
    analyticsData: null,
    metricsHistory: [],
    experiments: [],
    activeExperiment: null,
    notifications: [],

    setActiveCampaign: (campaign) => set({ activeCampaign: campaign }),
    setLoading: (isLoading) => set({ isLoading }),
    addNotification: (notification) =>
        set((state) => ({
            notifications: [notification, ...state.notifications],
            unreadCount: state.unreadCount + 1,
        })),
    dismissNotification: (id) =>
        set((state) => ({
            notifications: state.notifications.filter((n) => n.id !== id),

    appendLog: (log) =>
        set((state) => ({

        })),
```

## ⚙️ Backend API Deep Dive

The FastAPI backend serves as the **central orchestration supervisor** — accepting
requests from the frontend dashboard, routing intelligence tasks to the appropriate
MCP sub-processes via Stdio channels, aggregating responses, and returning unified
JSON payloads.

### 🔌 MCP Client Service

The `mcp_client.py` service manages the entire lifecycle of all four MCP sub-processes:

```python
# backend/app/services/mcp_client.py

import asyncio
import json
from typing import Any, Dict

class MCPClient:
    """
    Manages Stdio communication channels with all four MCP microservices.
    Each MCP process is launched as an isolated subprocess using asyncio
    subprocess management, communicating via StdIn/StdOut JSON-RPC messages.
    """

    def __init__(self, server_name: str, server_path: str):
        self.server_name = server_name
        self.server_path = server_path
        self.process: asyncio.subprocess.Process | None = None

    async def connect(self):
        """Launch the MCP subprocess and open Stdio communication pipes."""
        self.process = await asyncio.create_subprocess_exec(
            "node", self.server_path,
            stdin=asyncio.subprocess.PIPE,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE
        )
        print(f"🔌 Connected to MCP Server via Stdio: [{self.server_name}]...")

    async def call_tool(self, tool_name: str, arguments: Dict[str, Any]) -> Dict:
        """Send a tool invocation request to the MCP subprocess and await response."""
        request = {
            "jsonrpc": "2.0",
            "id": 1,
            "method": "tools/call",
            "params": {
                "name": tool_name,
                "arguments": arguments
            }
        }

        # Write request to subprocess StdIn
        self.process.stdin.write(
            json.dumps(request).encode() + b"\n"
        )
        await self.process.stdin.drain()

        # Read response from subprocess StdOut
        response_line = await self.process.stdout.readline()
        return json.loads(response_line.decode())

    async def disconnect(self):
        """Gracefully terminate the MCP subprocess."""
        if self.process:
            self.process.terminate()
            await self.process.wait()
```

**Symptom:**


**Root Cause:** The `useCampaignStore` hook is exported as a `default` export
rather than a named export, causing Webpack module resolution to fail.

**Remediation:**

```typescript
// ✅ CORRECT — Named export for Webpack compatibility
export const useCampaignStore = create<CampaignStore>((set) => ({
    campaigns: [],
    activeCampaign: null,
    isLoading: false,
    setCampaigns: (campaigns) => set({ campaigns }),
    setActiveCampaign: (campaign) => set({ activeCampaign: campaign }),

}));

export default create<CampaignStore>((set) => ({ ... }));
```

## 🤝 Contributing

Contributions to the AetherGrowth OS platform are welcome from authorized team
members and approved external collaborators. Please adhere to the following
contribution standards:

### Development Workflow

```bash
# 1. Fork the repository and clone your fork
git clone https://github.com/YOUR_USERNAME/AetherGrowth.git
cd AetherGrowth

# 2. Create a feature branch with a descriptive name
git checkout -b feature/add-websocket-support

# 3. Implement your changes following the existing code style
# 4. Test your changes thoroughly

# 5. Commit with a descriptive conventional commit message
git add .
git commit -m "feat(backend): add WebSocket endpoint for real-time dashboard updates"

# 6. Push and open a Pull Request against the main branch
git push origin feature/add-websocket-support
```

### Commit Message Convention

```
feat(scope):     New feature addition
fix(scope):      Bug fix
docs(scope):     Documentation update
refactor(scope): Code restructuring without behavior change
test(scope):     Test addition or modification
chore(scope):    Build, tooling, or dependency updates
```
